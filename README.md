# 人民法院案例库 · 民事案例知识库

## 一、数据源选择

### 1.1 数据源简介

[人民法院案例库](https://rmfyalk.court.gov.cn)（rmfyalk.court.gov.cn）是由最高人民法院建设并维护的官方案例数据库，于 2024 年 2 月 27 日正式上线并向社会开放。该库收录了经最高人民法院审核认可的指导性案例、参考案例及其他典型案例，是当前中国法律实务中权威性最高的案例数据源之一。

截至采集时（2026 年 6 月），案例库共收录案例约 5,445 篇，涵盖刑事、民事、行政、国家赔偿、执行五大类型，其中民事案例约 2,288 篇（参考案例 + 指导性案例）。

### 1.2 选择该数据源的理由

选择人民法院案例库作为民事案例知识库的数据源，基于以下考量：

**权威性优先。** 根据《人民法院案例库建设运行工作规程》（法〔2024〕92 号），入库案例由最高人民法院各审判业务庭室审核把关。入库参考案例的裁判要旨对类案审判具有参照效力，其权威性在现有公开案例资源中处于最高层级，符合知识库"权威性→合法性→可追溯性"的质量基线。

**结构化程度高。** 每个入库案例均包含入库编号、关键词、裁判要旨、基本案情、裁判理由、关联索引等标准化字段，正文结构按案例子类型（普通民事 / 执行 / 调解）统一编排，便于后续的法律检索与数据分析。

**民事案例覆盖面广。** 民事入库案例涵盖合同纠纷、侵权责任、公司纠纷、知识产权、婚姻家庭、建设工程等核心民商事领域，与刘佳馨律师主攻的房地产与建设工程、公司投融资并购、民商事争议解决等业务方向高度匹配。

### 1.3 与其他数据源的比较

| 数据源 | 权威性 | 结构化 | 全文可获取 | 登录要求 |
|--------|--------|--------|------------|----------|
| 人民法院案例库 | ★★★★★ 最高法审核 | ★★★★★ 标准化字段 | 是 | 需要（实名认证） |
| 中国裁判文书网 | ★★★★ 各级法院 | ★★ 仅有原始裁判文书 | 是 | 需要 |
| 最高院公报案例 | ★★★★ 最高法选编 | ★★★ | 部分 | 否 |
| 商业数据库（北大法宝等） | ★★★ | ★★★★ | 是 | 需要（付费） |

人民法院案例库在权威性与结构化程度上的综合优势使其成为本知识库的首选数据源。其缺陷在于总量尚有限（远少于裁判文书网的海量文书），且对自动化采集有登录认证屏障。

### 1.4 法律责任与使用限制

入库案例的收集和使用应当遵守《人民法院案例库建设运行工作规程》及最高人民法院相关版权声明。本知识库中的案例数据仅供个人法律研究之用，不得用于商业转售或大规模公开传播。

## 二、数据采集方法

### 2.1 技术架构概述

人民法院案例库采用前后端分离架构，数据通过 JSON API 动态加载。与传统 HTML 页面爬虫不同，本项目的采集流程直接与站点的 RESTful API 交互：

- **搜索 API**：`POST /cpws_al_api/api/cpwsAl/search` — 分页返回案例摘要列表
- **内容 API**：`POST /cpws_al_api/api/cpwsAl/content` — 根据案例 ID 返回完整正文
- **统计 API**：`POST /cpws_al_api/api/cpwsAl/indexTongji` — 返回全库案例总数（无需登录）

### 2.2 认证机制

案例库采用最高人民法院统一用户认证平台（account.court.gov.cn）进行 SSO 登录。用户的登录态通过以下方式传递：

- 浏览器端：OAuth 2.0 授权码流程（`HOLDONKEY` Cookie on account.court.gov.cn）
- API 端：`SESSION` Cookie（rmfyalk.court.gov.cn 域）

搜索 API 和内容 API 均要求携带有效的 `SESSION` Cookie，否则返回 `{"code": 401, "msg": "未登录"}`。

### 2.3 采集流程

本项目提供了两种采集路径，适用于不同场景：

#### 路径 A：Python 自动化脚本（日常增量更新）

脚本文件：[`scripts/scrape_rmfyalk_civil_cases.py`](../../脚本/scrape_rmfyalk_civil_cases.py)

```
用法：
  python3 scrape_rmfyalk_civil_cases.py           # 增量更新（默认）
  python3 scrape_rmfyalk_civil_cases.py --full     # 全量重新采集
  python3 scrape_rmfyalk_civil_cases.py --test     # 测试模式（仅 2 页）
```

**前置条件**：用户需先在浏览器中登录 rmfyalk.court.gov.cn，然后从开发者工具（F12 → Application → Cookies）中提取 `SESSION` 值，写入脚本同目录下的 `cookies.txt` 文件：

```
SESSION=<从浏览器复制的值>
```

或设置环境变量 `RMFYALK_SESSION`。

**采集逻辑**：

1. 调用搜索 API，参数 `case_sort_id_cpwsAl: "02"` 过滤民事案例，排序字段 `sort_field: "-cpws_al_zs_date"` 按裁判日期降序
2. 逐页获取案例摘要列表（每页最多 50 条）
3. 对每个案例调用内容 API 获取完整正文
4. 将 HTML 正文转换为 Markdown（使用 `markdownify` 库）
5. 按案例子类型（普通/执行/调解）自动区分正文模块结构
6. 增量更新：对比已保存的案例 ID，跳过已有案例

**请求控制**：每次请求间隔 0.3–0.45 秒（含随机抖动），API 调用失败自动重试 3 次（指数退避）。

#### 路径 B：Chrome DevTools Protocol 直采（初始批量采集）

当需要一次性批量采集但无法获取稳定 `SESSION` 时，使用 CDP（Chrome DevTools Protocol）直接从已登录的浏览器标签页注入 JavaScript 调用 API——自动携带浏览器原生的 Cookie 和认证上下文，无需手动复制 `SESSION`。

这是本知识库首批 20 个案例的实际采集方式。

### 2.4 采集数据字段映射

下表示出了 API 返回字段与 Markdown 输出字段之间的映射关系：

| API 字段 (`cpws_al_*`) | 输出位置 | 说明 |
|------------------------|----------|------|
| `title` / `no` | H1 标题、入库编号 | |
| `sub_title` | 副标题 | 裁判规则的简要概括 |
| `keyword` (array) | 关键词 | 逗号分隔 |
| `infos` | 案件信息行 | 复合字段，包含案号/法院/裁判日期/审理程序/入库日期 |
| `cpyz` | 裁判要旨 / 裁判要点 | 按案例类型自动命名 |
| `jbaq` | 基本案情 | |
| `cply` | 裁判理由 | 执行案件→执行理由，调解案件→调解结果 |
| `cpjg` | 裁判结果 | 仅指导性案例 |
| `glsy` | 关联索引 / 相关法条 | 法条引用 + 审级路径 |
| `ts_name` | 庭室信息 | |
| `update_time` | 文本调整时间 | |

### 2.5 输出目录

```

```

或本目录（`人民法院案例库知识库/`），文件命名格式：


## 三、数据清洗方法

### 3.1 噪音来源

从 API JSON 经过 `markdownify` 转换后，案例 Markdown 文件存在四类结构化噪音：

| 噪音类型 | 产生原因 | 表现 |
|----------|----------|------|
| 括号嵌套 | JSON 中庭室字段已含 `（民一庭）`，Markdown 生成时外层又包了一层 | `*（（民一庭））*` |
| 过度全角缩进 | 关联索引区域的层级缩进在执行过程中被加倍 | `　　　　一审：`（4 全角空格 → 应为 3） |
| 连续空行 | HTML 到 Markdown 转换残留的段落间距 | 3 行以上空白 → 应归一化为 2 行 |
| 零宽/隐形字符 | API 传输或编码转换过程中引入 | BOM、零宽空格、软连字符 |

### 3.2 清洗脚本

脚本文件：[`scripts/clean_rmfyalk_cases.py`](../../脚本/clean_rmfyalk_cases.py)

```
用法：
  python3 clean_rmfyalk_cases.py <目录>             # 预览模式
  python3 clean_rmfyalk_cases.py <目录> --apply      # 实际改写
```

#### 清洗规则

| 规则 | 正则/逻辑 | 示例 |
|------|-----------|------|
| 修复括号嵌套 | `\*（（([^）]+)））\*` → `*（\1）*` | `*（（民一庭））*` → `*（民一庭）*` |
| 缩减过度缩进 | 行首 4+ 全角空格 → 3 全角空格 | `　　　　一审：` → `　　　一审：` |
| 空行归一化 | `\n{3,}` → `\n\n` | |
| 末尾空行修剪 | `rstrip("\n")` + 追加恰好 1 行 | |
| 零宽字符清除 | 删除 `﻿` `​` `‌` `‍` `­` `‎` `‏` | |

## 四、脱敏处理方法

### 4.1 脱敏原理

人民法院案例库中的案例文本在入库时已经做了初步脱敏——自然人姓名以"某"字替代全名或名的一部分，公司名称中的字号以"某"字替代。因此文本中的原始实体名已具备如下模式：

- **公司/机构名**：含"某"的全称或简称 + 公司后缀（有限/责任/股份/集团/银行/中心…）
  - 例：`四川鸿某实业集团有限公司`、`茂某建材公司`
- **自然人姓名**：1–2 个汉字 + "某" + 0–2 个汉字
  - 例：`呙某元`、`孙某`、`李某某`

本项目的脱敏目标是在已脱敏文本的基础上，将各主体名称统一替换为**无信息量的通用代号**，确保：
1. 同一主体的所有出现被替换为同一代号（可追溯性）
2. 不同主体的代号不可互推（不可关联性）
3. 法院、检察院名称不被替换（权威机构名称需保留）

### 4.2 脱敏脚本

脚本文件：[`scripts/anonymize_rmfyalk_cases.py`](../../脚本/anonymize_rmfyalk_cases.py)

```
用法：
  python3 anonymize_rmfyalk_cases.py <目录>                # 预览映射表
  python3 anonymize_rmfyalk_cases.py <目录> --apply         # 实际改写
  python3 anonymize_rmfyalk_cases.py <目录> --apply --dry   # 输出到 .anonymized.md
```

#### 实体识别策略

脱敏脚本采用"高信噪比源优先"的三步识别策略：

**Step 1：案件标题解析。** 案例标题遵循固定格式——`原告诉被告及第三人XX纠纷案`。脚本以"诉""与""及""、"为分隔符切分当事人列表，从每个分片中提取含"某"的实体名。按后缀特征分为公司（含银行/公司/中心/保险/道馆……）和自然人（2–5 汉字含某）。

```
"呙某元诉四川鸿某实业集团有限公司及第三人绵阳茂某建材有限公司债权人代位权纠纷案"
 → ["呙某元", "四川鸿某实业集团有限公司", "绵阳茂某建材有限公司"]
```

**Step 2："以下简称"反向定位。** 案例正文中大量使用"全称（以下简称简称）"构造来定义当事人简称。脚本以正则匹配 `[（(]以下简称\s*([^）)]{2,20})\s*[）)]` 定位简称，再向前 100 字符窗口内搜索最近的"含某 + 公司后缀"匹配作为对应的全称。

```
"呙某元与绵阳茂某建材有限公司（以下简称茂某建材公司）"
 → full="绵阳茂某建材有限公司", alias="茂某建材公司"
```

**Step 3：正文补漏。** 对"以下简称"未覆盖的实体，以公司后缀（有限/责任/股份/集团/银行/中心……）为锚点，在紧邻词边界（标点/行首/连接词后，标点/行尾/连接词前）的约束下，用正则匹配余下的公司名和自然人姓名。

#### 标签体系

- **公司/机构**：天干 + 地支 + "公司"（120 种组合，如 `甲子公司`、`乙丑公司`……`癸亥公司`），溢出用数字下标（`公司121`）
- **自然人**：天干 + "某"（10 种组合，如 `甲某`、`乙某`……`癸某`），溢出用数字下标（`当事人11`）

标签按实体出现频率降序分配——高频实体获得更短的标签。

#### 替换策略

长名优先替换：将所有实体名按字符串长度降序排列后依次替换，避免较短的简称在较长的全称中被先行替换导致全称残留（如"茂某建材公司"→"乙公司"后，"绵阳茂某建材有限公司"的前半部分错误残留）。

### 4.3 脱敏范围与保留项

| 类别 | 处理 | 示例 |
|------|------|------|
| 公司全称 | ✓ 替换 | `四川鸿某实业集团有限公司` → `甲子公司` |
| 公司简称 | ✓ 替换 | `鸿某实业公司` → `甲子公司` |
| 自然人姓名 | ✓ 替换 | `呙某元` → `甲某` |
| 法院名称 | ✗ 保留 | `最高人民法院`、`乌海市乌达区人民法院` |
| 检察院名称 | ✗ 保留 | `马鞍山市人民检察院` |
| 法条引用 | ✗ 保留 | `《中华人民共和国民法典》第1064条` |
| 案号 | ✗ 保留 | `（2022）内0304民初932号` |
| 裁判日期 | ✗ 保留 | `2022.09.15` |

### 4.4 映射对照表

每次脱敏执行后自动生成 `_anonymize_map.txt` 文件，格式为：

```
甲子公司 ← 四川鸿某实业集团有限公司
甲子公司 ← 鸿某实业公司
甲某 ← 呙某元
乙某 ← 孙某
```

该文件是脱敏后的案例与原始数据之间唯一的可逆对照记录，应妥善保管。

## 五、文件格式总览

每个案例 Markdown 文件包含两个区域：

### 5.1 YAML Front Matter

```yaml
---
title: "脱敏后的案件标题"
case_no: 2026-07-2-108-001
case_type: 参考案例
subtitle: "裁判规则一句话概括"
case_category: 民事
cause_of_action: 借款合同纠纷
court: 乌海市乌达区人民法院
judgment_date: 2022.09.15
case_docket: (2022)内0304民初932号
procedure: 一审
entry_date: 2026.05.20
keywords:
  - "民事"
  - "借款合同"
  - ...
status: 有效
trial_division: 民一庭
source: ["最高人民法院"]
source_url: "https://rmfyalk.court.gov.cn/view/content.html?id=..."
collected_at: "2026-06-26 17:30:38"
---
```

### 5.2 正文

按案例子类型自动区分的模块化结构：

- **参考案例（普通）**：基本案情 → 裁判理由 → 裁判要旨 → 关联索引
- **参考案例（执行）**：基本案情 → 执行理由 → 执行要旨 → 关联索引
- **参考案例（调解）**：基本案情 → 调解结果 → 调解指引 → 案例价值 → 关联索引
- **指导性案例**：裁判要点 → 基本案情 → 裁判结果 → 裁判理由 → 相关法条

正文末尾附庭室信息（如 `*（民一庭）*`）及指导性案例的文本声明。

## 七、后续改进方向

- **实体识别精度**：当前标题 + "以下简称"的识别策略覆盖约 80–90% 的主体名称。少量仅出现在正文中的简称（如"奥某科技公司"）未被捕捉。后续可引入分词工具（jieba）辅助边界检测，或对脱敏结果执行第二遍补漏扫描。
- **案例子类型扩展**：当前支持的子类型包括普通参考案例、执行案件、调解案件、指导性案例。特色案事例（`cpws_al_type: "04"`）及未成年人保护案例的正文结构未完整适配。
- **OCR 法条引用**：关联索引中的法条引用为 HTML 格式，部分嵌入的案号索引格式不规范，可考虑用正则标准化为结构化引用。
- **增量采集自动化**：当前脚本需要手动更新 SESSION Cookie。可通过搭建定时登录续期的本地代理层实现全自动化增量更新。
